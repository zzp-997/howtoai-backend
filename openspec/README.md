# OpenSpec 框架

OpenSpec是一套结构化的变更管理工具，旨在规范化软件开发过程中的需求、设计和实现。

## 目录结构

```
openspec/
├── changes/          # 变更工件
│   ├── <change-name>/ # 具体变更目录
│   └── archive/      # 已完成的变更归档
├── specs/           # 主规范文件
├── config.yaml      # OpenSpec配置
└── README.md        # 本文档
```

## 核心工作流

OpenSpec遵循四个核心阶段的生命周期：

### 1. 探索 (`/opsx:explore`)
- 用于思考和分析，而不是实施代码
- 探索问题空间、挑战假设、可视化解决方案
- 可以创建OpenSpec工件（提案、设计、规格），但不实现功能
- 使用ASCII图表来可视化系统架构和流程

#### 使用场景：
- 开始复杂功能前的需求分析
- 解决技术难题
- 方案权衡和决策

### 2. 提议 (`/opsx:propose`)
- 创建变更提案，包括所有必要工件
- 生成proposal.md（什么和为什么）、design.md（如何）、tasks.md（实施步骤）
- 使用kebab-case命名变更（例如：`add-user-authentication`）

```bash
/opsx:propose add-user-authentication
```

### 3. 实施 (`/opsx:apply`)
- 根据tasks.md逐步实施功能
- 保持每次更改最小且专注
- 实施期间持续更新任务状态（- [ ] → - [x]）
- 在实施过程中发现问题时，可以更新设计工件

### 4. 归档 (`/opsx:archive`)
- 在完成所有任务后归档变更
- 清理工作空间，将变更存档
- 同步增量规范到主规范（如果需要）

## 最佳实践原则

### 分阶段执行
- 不要跳过探索阶段直接实施
- 完成提案后再开始实施
- 确保所有任务完成后再归档

### 工件完整性
- **Proposal**：清楚描述变更的动机和范围
- **Design**：详细说明实现方案和架构决策
- **Tasks**：将实现分解为具体的、可执行的步骤

### 代码变更控制
- 保持每个任务的代码变更最小
- 专注于完成一个任务再继续下一个
- 避免在任务外做额外的修改

### 状态管理
- 定期检查变更状态：`openspec status --change <name>`
- 确保所有工件都处于完成状态
- 在归档前确认所有任务都已完成

## 命令使用指南

### 探索模式 (`/opsx:explore`)
- 在开始复杂功能前使用
- 不编写代码，只分析和设计
- 用于解决复杂问题或权衡方案
- 保持开放性思维，不要急于得出结论

### 提议模式 (`/opsx:propose`)
- 用于创建结构化变更
- 自动生成工件，遵循模板
- 基于项目上下文和约束生成设计
- 确保所有依赖的工件都被正确创建

### 应用模式 (`/opsx:apply`)
- 遵循任务清单逐步实现
- 在遇到障碍时暂停并寻求指导
- 实施过程中如发现设计问题，可返回更新设计
- 每完成一个任务立即标记为完成

### 归档模式 (`/opsx:archive`)
- 只在所有任务完成后执行
- 确认没有未完成的工件或任务
- 同步规范更新到主规范系统
- 创建清晰的变更历史记录

## 探索模式限制

- 不得编写应用代码或实现功能
- 可以创建OpenSpec工件来记录决策
- 如用户要求实现，需提醒先退出探索模式

## 工件一致性

- 确保Proposal、Design和Tasks之间的一致性
- 设计变更需反映在所有相关工件中
- 保持工件与实际实现同步

## 故障排除

### 当实施受阻时
- 返回到探索模式分析问题
- 可能需要更新设计或任务
- 与团队沟通复杂的技术挑战

### 工件不一致时
- 检查工件之间的依赖关系
- 确保所有上下文文件都已读取
- 遵循工件创建的正确顺序

### 变更复杂度过高
- 将大变更拆分为多个小变更
- 每个变更聚焦单一功能点
- 避免跨多个功能领域的复杂变更

## 状态检查命令

常用命令用于检查OpenSpec变更的状态：

```bash
# 查看所有变更
openspec list

# 查看特定变更状态
openspec status --change <change-name>

# 以JSON格式查看变更状态
openspec status --change <change-name> --json
```

OpenSpec的目标是提供一种系统化的方法来管理和实现变更，确保高质量的代码交付，同时维护清晰的变更历史和设计决策记录。
